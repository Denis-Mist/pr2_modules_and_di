# Практическое занятие 2 — Система расширяемых модулей с DI

## Быстрый старт

### Требования
- .NET 8 SDK (`dotnet --version` ≥ 8.0)

### Запуск приложения
```bash
cd csharp/src/Pr2.ModulesAndDi
dotnet run
```

### Запуск тестов
```bash
cd csharp
dotnet test
```

### Запуск с другим набором модулей
Отредактируйте `appsettings.json` — перечислите нужные модули в секции `"Modules"`.

Пример с минимальным набором (только Core и Logging):
```json
{
  "Modules": ["Core", "Logging"]
}
```

Пример без модуля Export (Report зависит от Export — будет ошибка):
```json
{
  "Modules": ["Core", "Validation", "Report"]
}
```
Приложение завершится с понятным сообщением:  
`Не хватает модуля для зависимости: модуль 'Report' требует 'Export'...`

## Разграничение ядра и модулей

### Ядро (`Core/`)
- Определяет **контракт** `IAppModule` (имя, зависимости, `RegisterServices`, `InitializeAsync`)
- `ModuleCatalog`: обнаруживает модули через отражение, строит порядок запуска (алгоритм Кана)
- `Program.cs`: оркестрирует жизненный цикл — читает конфиг, запускает модули, вызывает действия
- **Ядро не импортирует пространства имён конкретных модулей**

### Модули (`Modules/`)
- Реализуют `IAppModule`, регистрируют свои службы и действия в DI-контейнере
- Могут зависеть от других модулей через `Requires` — ядро гарантирует порядок запуска
- **Добавление нового модуля не требует изменения ядра**: достаточно создать класс, реализующий `IAppModule`, и добавить имя в `appsettings.json`

## Граф зависимостей модулей

```
Core
├── Logging      (Core)
├── Notification (Core)
├── Validation   (Core)
│   └── Export   (Core, Validation)
│       └── Report (Core, Export)
```

Порядок запуска, построенный топологической сортировкой:  
`Core → Logging → Notification → Validation → Export → Report`

---

## Внедрение зависимостей: два времени жизни

| Время жизни | Тип | Поведение |
|-------------|-----|-----------|
| **Singleton** | `IClock`, `IStorage`, `IEventBus` | Один экземпляр на всё приложение. Все модули работают с одним хранилищем и одной шиной. |
| **Transient** | `TransientRequestContext` | Новый экземпляр при каждом обращении к контейнеру. `LoggingAction` получает два разных `ctx1` и `ctx2` — это видно в выводе (разные `Id`). |

**Доказательство в тестах**: `Singleton_возвращает_один_и_тот_же_экземпляр` и `Transient_возвращает_разные_экземпляры`.

## Сценарии проверки (appsettings.json)

### Сценарий 1: полный набор модулей
```json
{ "Modules": ["Core","Logging","Notification","Validation","Export","Report"] }
```
Все модули загружаются, данные проверяются → экспортируются → попадают в отчёт.

### Сценарий 2: отсутствующий модуль
```json
{ "Modules": ["Core","Export"] }
```
Ожидаемый вывод:
```
[ОШИБКА МОДУЛЯ] Не хватает модуля для зависимости: модуль 'Export' требует 'Validation'...
```

### Сценарий 3: только Core и Logging
```json
{ "Modules": ["Core","Logging"] }
```
Минимальное приложение — только журналирование и демонстрация времён жизни DI.

### Сценарий 4: только Core
```json
{ "Modules": ["Core"] }
```
Инфраструктура запускается, действий нет.

| 1 | Порядок запуска при линейной цепочке зависимостей A→B→C |
| 2 | Порядок при ветвлении: Core, Logging, Validation, Export |

| 6 | Сообщение об ошибке содержит имя недостающего модуля |
| 7 | Недостающая транзитивная зависимость — понятное сообщение |
| 8 | Прямой цикл двух модулей обнаруживается |
| 9 | Длинный цикл трёх модулей (A→B→C→A) обнаруживается |
| 10 | Сообщение о цикле содержит имена проблемных модулей |
| 11 | Зависимости внедряются из контейнера (не создаются вручную) |
| 12 | Singleton — один экземпляр при двух обращениях |
| 13 | Transient — разные экземпляры при двух обращениях |
| 14 | Несовместимая версия контракта — понятная ошибка |
| 15 | Совместимая версия контракта — загрузка успешна |
